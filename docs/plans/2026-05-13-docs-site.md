# Documentation Site (Single-Page) Implementation Plan

> **REQUIRED SUB-SKILL:** Use the executing-plans skill to implement this plan task-by-task.

**Goal:** Replace the scattered `README.md` + `doc/guides/*.md` + per-example `README.md` set with a single long-scroll documentation site (`docs/index.html`) that interleaves prose with code samples and live Pixi previews of each example, in the spirit of pixijs.com / vite.dev — fulfilling TODO §11 and revisiting §9 so examples and docs share one visual style.

**Architecture:**
- Hand-author one long `docs/index.html` with a sticky sidebar nav, scroll-spy, and section anchors. No bundler.
- Use a `<script type="importmap">` to load Pixi from `esm.sh` and the compiled library + examples from `../dist/`. Each example section embeds a `<canvas>` mounted by a `docs/bootstrap.ts` that calls a new `mountDemo(target)` export added to each example.
- Keep `examples/*/demo.ts` exactly as-is (Node-safe `run...Demo()` functions still drive `test/examples.test.ts`); add a new browser-only entry `examples/*/preview.ts` per example with a `mountDemo(target)` export.
- Keep `doc/pxd-v1.md` as the canonical spec; collapse `doc/guides/*.md` into the new page (replace each guide file with a short stub pointing at the site to avoid leaving stale duplicates).
- A new `test/docs.test.ts` asserts: `docs/index.html` exists, contains every required section id, embeds a canvas for every example, and the `mountDemo` export exists in every example `preview.ts`.

**Tech Stack:** Plain HTML/CSS/ES modules, importmap → `esm.sh/pixi.js@8`, Prism via CDN for code highlighting, TypeScript (existing `tsc` pipeline) for `bootstrap.ts` and `preview.ts`, Node `node:test` for verification.

## Codebase analysis summary

- Public API frozen: `build` / `apply` / `find` / `findAll` / `requirePath` / `getSlot` / `mountSlot` / `validate` / `defaultNodeTypes` / `mergeRegistry` / `resolveDecisionValue` / `resolveBindings` (`src/index.ts`).
- Examples already compile through `tsconfig.json` (root `include: ["src/**/*.ts", "test/**/*.ts", "examples/**/*.ts"]`) and emit to `dist/examples/...`. Adding `examples/*/preview.ts` requires no new tsc config.
- `package.json.files` currently ships `examples/`, `dist/src`, `dist/examples`, `doc/pxd-v1.md`, `pxd.schema.json`, `README.md`, `LICENSE`. We will add `docs/index.html` + `docs/bootstrap.js` (built from `docs/bootstrap.ts`) + `docs/styles.css` + `docs/site.js` to `files` so npm consumers can open `node_modules/pxd/docs/index.html` directly.
- `tsconfig.json` `include` must additionally cover `docs/**/*.ts` so `bootstrap.ts` compiles to `dist/docs/bootstrap.js`. The HTML imports it via a relative path resolved against the doc page's deployed location. To avoid two source/dist trees in the user's mental model we will emit `docs/bootstrap.js` next to `docs/index.html` by adding a separate small `tsconfig.docs.json` that targets `docs/`. Decision: simpler is a separate compile step. We'll add `tsconfig.docs.json` with `outDir: "docs"` and `rootDir: "docs"` and a `docs:build` npm script.
- Pixi v8 supports ESM out of the box via `esm.sh/pixi.js@8.7`. Importmap entries: `"pixi.js": "https://esm.sh/pixi.js@8.7"`, `"pxd": "../dist/src/index.js"`, and one per example: `"@example/<name>": "../dist/examples/<name>/preview.js"`. The bootstrap dynamically imports via these bare specifiers.
- The browser-minimal example currently has `index.html` + `main.ts`. We will keep that folder but make `main.ts` re-export `mountDemo` from the new shared `preview.ts` so the standalone HTML and the docs site share code.
- TODO §9 is marked complete; revisiting means consolidating its visual layer (every example gains a real browser preview rather than only the Node smoke export). TODO §11 is the new bulk of the work.
- The custom composable example, decisions/bindings example, and hot-reload example need a small Pixi visual representation in their `preview.ts` (the existing `runFooDemo()` only returns objects). Each preview is small and self-contained — no shared chrome required.

What this plan deliberately does NOT do:
- No frontend framework, no Vite, no markdown→HTML generator, no Prism build step (CDN only).
- No design-system extraction; styles live inline in `docs/styles.css`.
- No multi-page docs router; one page, anchor links only.
- No `mode: "full"`/reconcile/extensions material — the page must reflect that these are not implemented (§11 TODO bullet).
- No deletion of `doc/pxd-v1.md` (canonical spec stays).

import { Application, Assets, Texture } from "pixi.js";
import { build } from "../../src/index.js";
import { browserDoc } from "./document.js";

export async function mountDemo(target: HTMLElement): Promise<void> {
    const app = new Application();
    await app.init({ background: "#0f172a", width: 520, height: 280, antialias: true });
    target.replaceChildren(app.canvas);

    const root = build(browserDoc, {
        resolve: {
            texture: (id) => Assets.get(id) ?? Texture.WHITE,
            style: (id) => {
                if (id === "title") return { fill: "#f8fafc", fontSize: 28, fontFamily: "Inter, Arial", fontWeight: "700" };
                if (id === "caption") return { fill: "#cbd5e1", fontSize: 14, fontFamily: "Inter, Arial", wordWrap: true };
                return undefined;
            },
        },
    });
    app.stage.addChild(root);
}

**Step 2: Create `docs/site.js`** (scrollspy + smooth anchors, no framework)

```js
const sectionLinks = [...document.querySelectorAll("aside.sidebar a[href^='#']")];
const sectionTargets = sectionLinks
    .map((a) => document.getElementById(a.getAttribute("href").slice(1)))
    .filter(Boolean);

const linkById = new Map();
for (const a of sectionLinks) linkById.set(a.getAttribute("href").slice(1), a);

const observer = new IntersectionObserver((entries) => {
    for (const entry of entries) {
        if (!entry.isIntersecting) continue;
        for (const a of sectionLinks) a.classList.remove("active");
        const link = linkById.get(entry.target.id);
        if (link) link.classList.add("active");
    }
}, { rootMargin: "-30% 0% -60% 0%", threshold: 0 });

for (const t of sectionTargets) observer.observe(t);

document.addEventListener("click", (e) => {
    const target = e.target;
    if (!(target instanceof HTMLAnchorElement)) return;
    const href = target.getAttribute("href");
    if (!href || !href.startsWith("#")) return;
    const el = document.getElementById(href.slice(1));
    if (!el) return;
    e.preventDefault();
    el.scrollIntoView({ behavior: "smooth", block: "start" });
    history.replaceState(null, "", href);
});
```

    <h2>Custom node types</h2>
    <p>The default registry covers intrinsics: <code>container</code>, <code>sprite</code>, <code>nineSliceSprite</code>, <code>tilingSprite</code>, <code>animatedSprite</code>, <code>text</code>, <code>bitmapText</code>, <code>graphics</code>, <code>slot</code>.     <p>Spine and other engine-specific objects are intentionally not built-ins. Register them as custom <code>nodeTypes</code> so your engine owns the Spine runtime dependency and field semantics.</p>

    <h3>The contract</h3>
    <pre><code class="language-ts">interface NodeType {
    create(node: ResolvedNode, ctx: BuildContext): Container;
    assign?(node: ResolvedNode, target: Container, ctx: AssignContext): void;
}</code></pre>
    <ul>
        <li><code>create</code> constructs an empty container subclass.</li>
        <li><code>assign</code> writes type-specific fields. Called by both <code>build</code> and <code>apply</code> — write your field-application logic once.</li>
    </ul>

    <h3>Example: SpinButton</h3>
    <pre><code class="language-ts">import { build, defaultNodeTypes, type NodeType } from "pxd";
import { Container, Text } from "pixi.js";

class SpinButton extends Container {
    private caption = new Text({ text: "" });
    constructor() { super(); this.addChild(this.caption); this.eventMode = "static"; }
    setLabel(s: string) { this.caption.text = s; }
    setEnabled(on: boolean) { this.alpha = on ? 1 : 0.4; this.eventMode = on ? "static" : "none"; }
}

const SpinButtonType: NodeType = {
    create: () => new SpinButton(),
    assign: (node, target, ctx) => {
        if (!(target instanceof SpinButton)) return;
        if (typeof node.text === "string") target.setLabel(ctx.readString(node.text));
        if (typeof node.enabled === "boolean") target.setEnabled(node.enabled);
    },
};

build(doc, {
    resolve: { texture: Assets.get },
    nodeTypes: new Map([...defaultNodeTypes, ["SpinButton", SpinButtonType]]),
});</code></pre>

    <h3>Custom fields live on the node</h3>
    <p>There is no <code>props</code> container. Custom scalar fields sit at the top level and participate in decision resolution.</p>
    <pre><code class="language-json">{
    "id": "spinBtn",
    "type": "SpinButton",
    "x": 600, "y": 540,
    "text": { "_": "SPIN", "mobile": "TAP" },

}</code></pre>

## Implementation notes and guardrails

- Keep importmap entries up to date if you add or rename an example — both the HTML `<script type="importmap">` and the `<script type="module">` import block at the bottom of the page must be edited together.
- Do not introduce a bundler. Importmap + `esm.sh` is the only delivery mechanism.
- Prism CDN is for syntax highlighting only — the page must render correctly without it (the JS autoloader is failure-tolerant).
- `mountDemo(target)` must always replace `target`'s children (`target.replaceChildren(app.canvas)`) so re-running the docs page (e.g. HMR in a downstream consumer) does not stack canvases.
- For Node-side tests, do not import `preview.ts` modules eagerly at top level — they construct `Application` only inside `mountDemo`, so dynamic import in `docs.test.ts` is safe.
- Style guide alignment: prose voice in `docs/index.html` matches the existing guides — terse, second-person, no marketing language. Spec citations (e.g. `§3.6`, `§7.2`) are welcome where they help the reader cross-reference `doc/pxd-v1.md`.
- TODO §11 explicitly forbids promising `mode: "full"`, reconcile, or extensions in docs. Re-read the "What's not included" section before merging if anything tempts you to add a "coming soon" line.
- The example sections of the site supersede the per-folder example READMEs for end-users; the READMEs in `examples/README.md` and any per-example README can be slimmed but should still exist as a fallback index inside the npm tarball.
