Overview

pxd is a tiny Pixi.js library that reads PXD v1 documents (Core + Library), builds Container trees, walks them by label, and applies updated documents to live trees as patches.

PXD separates scene configuration (positions, textures, styles, prefabs) from scene logic (state, animations, reactions). This library is the Pixi-side bridge: hand it a JSON document, get a Pixi tree back; hand it an updated JSON document, get a live in-place patch.

Spec: doc/pxd-v1.md. The library implements Core and Library levels; Scene-shape documents are rejected.

Mental model

The public surface is intentionally small:

  • build(doc, options) — make a fresh Pixi tree from a document.
  • apply(doc, root, options) — patch an existing Pixi tree from a document.
  • find(root, "a.b.c") / findAll / requirePath — dot-path lookup by Container.label.
  • getSlot(root, name) / mountSlot — symbol-tagged mount points for host-owned content.
  • validate(doc) — semantic check (called automatically by build).

Everything else is an extension hook, not a separate concept. There is exactly one extension mechanism: the nodeTypes registry of { create, assign } strategies. Both build and apply use the same registry, so a single assign handles the initial build and any subsequent hot reloads. There are no extension handlers, no extension API, no reactivity — just one map.

Dispatch order per node: create → assign (type-specific fields) → base fields (x, y, scale, …). Base fields run last, so explicit transforms always override side effects of assign.

Quickstart

Install:

npm install pxd pixi.js

Minimal document:

{
    "format": "pxd",
    "version": 1,
    "root": {
        "id": "root",
        "type": "container",
        "children": [
            { "id": "title", "type": "text", "text": "Hello", "x": 100, "y": 40 },
            { "id": "logo",  "type": "sprite", "texture": "logo_main" }
        ]
    }
}

Build a tree:

import { Application, Assets, Text } from "pixi.js";
import { build, requirePath } from "pxd";
import doc from "./hud.json";

const app = new Application();
await app.init({ resizeTo: window });
document.body.appendChild(app.canvas);

const root = build(doc, {
    resolve: { texture: (id) => Assets.get(id) },
});
app.stage.addChild(root);

const title = requirePath<Text>(root, "title");
title.text = "Hello, player";

Container.label on every built node equals node.label ?? node.id — that's what find and requirePath match against.

Public API

Every export from pxd:

ExportKindPurpose
build(doc, options)functionValidate doc, construct a fresh Pixi Container tree.
apply(doc, root, options?)functionPatch an existing tree in place. Returns the number of updated nodes.
find(root, path)functionDot-path lookup by Container.label, returns null on miss.
findAll(root, path)functionAll matches at the final segment.
requirePath(root, path)functionLike find but throws on miss.
getSlot(root, name)functionLocate a slot Container by its slot field (uses Symbol.for("pxd.slot")).
mountSlot(root, name, child)functionFind slot, addChild(child); throws if slot is missing.
validate(doc)functionRun semantic validation. build calls this internally.
defaultNodeTypesMapDefault NodeType registry: container, sprite, nineSliceSprite, tilingSprite, animatedSprite, text, bitmapText, graphics, slot.
mergeRegistry(a, b)functionHelper for composing nodeTypes maps.
NodeType, BuildContext, AssignContext, Resolvers, BuildOptions, ApplyOptions, PxdDocument, …typesTypeScript-only.

Build options:

interface BuildOptions {
    resolve: {
        texture(id: string): Texture | undefined;
        style?(id: string): Partial<TextStyleOptions> | undefined;
        binding?(path: string): string | number | boolean | undefined;
    };
    activeTags?: ReadonlyArray<string>;
    nodeTypes?: Map<string, NodeType>;  // defaults to defaultNodeTypes
}

interface ApplyOptions extends Partial<BuildOptions> {
    onMissing?(path: string, nodeId: string): void;
}

Reserved base fields

The library reads these fields off every node, intrinsic or custom. Do not reuse the names for custom semantics — they will be overwritten by the base-field pass that runs last in the per-node dispatch order.

FieldEffect
idIdentifier within the document (and within a prefab scope). Used by mask references.
typeIntrinsic key or custom NodeType key.
labelOptional friendly name. Container.label = node.label ?? node.id. Path lookup uses this.
x, yPosition.
scaleX, scaleYScale on each axis.
rotationRotation in radians.
alpha, visible, zIndexPixi base props.
pivotX, pivotYPivot offset.
maskId reference to another node used as the live target's Pixi mask.
childrenStructural. The library builds and walks these; your assign must not touch them.
extensionsReserved. Per-node payloads are silently ignored in this version.
Custom fields are everything else. Any other top-level field on a custom or runtime node is your responsibility: the library will run it through decision resolution (§3.6) and binding substitution (§7.2) and pass the resolved value to your NodeType.assign. There is no schema check for custom field shapes — validate them inside your assign or in a host pre-flight step.

apply() patch semantics

apply() is patch-only. It mutates fields that are present in the apply document and leaves everything else untouched. There is no mode: "full" today.

SituationBehaviour
Live node has x: 100, apply doc omits xLive x stays 100.
Live node has a mask, apply doc omits maskMask stays attached.
Apply doc has a child not in the live treeonMissing(path, id) fires; subtree is skipped.
Live tree has a child not in the apply docLive child stays attached unmodified.
Type mismatch (doc says text, live target is Sprite)Type-specific fields skipped silently. Base fields still flow.
Renamed descendant label in apply docTreated as a missing child (path key changed). Use rebuild for structural renames.

Matching uses one-hop child lookup against Container.label at each step — not deep search. Masks rebind by id against the live tree's id map; the source may live outside the apply doc.
