Overview

pxd is a tiny Pixi.js library that reads PXD v1 documents (Core + Library), builds Container trees, walks them by label, and applies updated documents to live trees as patches.

PXD separates scene configuration (positions, textures, styles, prefabs) from scene logic (state, animations, reactions). This library is the Pixi-side bridge: hand it a JSON document, get a Pixi tree back; hand it an updated JSON document, get a live in-place patch.

Spec: doc/pxd-v1.md. The library implements Core and Library levels; Scene-shape documents are rejected.

Mental model

The public surface is intentionally small:

  • build(doc, options) — make a fresh Pixi tree from a document.
  • apply(doc, root, options) — patch an existing Pixi tree from a document.
  • find(root, "a.b.c") / findAll / requirePath — dot-path lookup by Container.label.
  • getSlot(root, name) / mountSlot — symbol-tagged mount points for host-owned content.
  • validate(doc) — semantic check (called automatically by build).

Everything else is an extension hook, not a separate concept. There is exactly one extension mechanism: the nodeTypes registry of { create, assign } strategies. Both build and apply use the same registry, so a single assign handles the initial build and any subsequent hot reloads. There are no extension handlers, no extension API, no reactivity — just one map.

Dispatch order per node: create → assign (type-specific fields) → base fields (x, y, scale, …). Base fields run last, so explicit transforms always override side effects of assign.

Quickstart

Install:

npm install pxd pixi.js

Minimal document:

{
    "format": "pxd",
    "version": 1,
    "root": {
        "id": "root",
        "type": "container",
        "children": [
            { "id": "title", "type": "text", "text": "Hello", "x": 100, "y": 40 },
            { "id": "logo",  "type": "sprite", "texture": "logo_main" }
        ]
    }
}

Build a tree:

import { Application, Assets, Text } from "pixi.js";
import { build, requirePath } from "pxd";
import doc from "./hud.json";

const app = new Application();
await app.init({ resizeTo: window });
document.body.appendChild(app.canvas);

const root = build(doc, {
    resolve: { texture: (id) => Assets.get(id) },
});
app.stage.addChild(root);

const title = requirePath<Text>(root, "title");
title.text = "Hello, player";

Container.label on every built node equals node.label ?? node.id — that's what find and requirePath match against.

Public API

Every export from pxd:

ExportKindPurpose
build(doc, options)functionValidate doc, construct a fresh Pixi Container tree.
apply(doc, root, options?)functionPatch an existing tree in place. Returns the number of updated nodes.
find(root, path)functionDot-path lookup by Container.label, returns null on miss.
findAll(root, path)functionAll matches at the final segment.

defaultNodeTypesMapDefault NodeType registry: container, sprite, nineSliceSprite, tilingSprite, animatedSprite, text, bitmapText, graphics, slot.

NodeType, BuildContext, AssignContext, Resolvers, BuildOptions, ApplyOptions, PxdDocument, …typesTypeScript-only.

Build options:

interface BuildOptions {
    resolve: {
        texture(id: string): Texture | undefined;
        style?(id: string): Partial<TextStyleOptions> | undefined;
        binding?(path: string): string | number | boolean | undefined;
    };
    activeTags?: ReadonlyArray<string>;
    nodeTypes?: Map<string, NodeType>;  // defaults to defaultNodeTypes
}

interface ApplyOptions extends Partial<BuildOptions> {
    onMissing?(path: string, nodeId: string): void;
}

Reserved base fields

The library reads these fields off every node, intrinsic or custom. Do not reuse the names for custom semantics — they will be overwritten by the base-field pass that runs last in the per-node dispatch order.

FieldEffect
idIdentifier within the document (and within a prefab scope). Used by mask references.
typeIntrinsic key or custom NodeType key.
labelOptional friendly name. Container.label = node.label ?? node.id. Path lookup uses this.
x, yPosition.
scaleX, scaleYScale on each axis.
rotationRotation in radians.
alpha, visible, zIndexPixi base props.
pivotX, pivotYPivot offset.
maskId reference to another node used as the live target's Pixi mask.
childrenStructural. The library builds and walks these; your assign must not touch them.
extensionsReserved. Per-node payloads are silently ignored in this version.
Custom fields are everything else. Any other top-level field on a custom or runtime node is your responsibility: the library will run it through decision resolution (§3.6) and binding substitution (§7.2) and pass the resolved value to your NodeType.assign. There is no schema check for custom field shapes — validate them inside your assign or in a host pre-flight step.

apply() patch semantics

apply() is patch-only. It mutates fields that are present in the apply document and leaves everything else untouched. There is no mode: "full" today.

SituationBehaviour
Live node has x: 100, apply doc omits xLive x stays 100.
Live node has a mask, apply doc omits maskMask stays attached.
Apply doc has a child not in the live treeonMissing(path, id) fires; subtree is skipped.
Live tree has a child not in the apply docLive child stays attached unmodified.
Type mismatch (doc says text, live target is Sprite)Type-specific fields skipped silently. Base fields still flow.
Renamed descendant label in apply docTreated as a missing child (path key changed). Use rebuild for structural renames.

Matching uses one-hop child lookup against Container.label at each step — not deep search. Masks rebind by id against the live tree's id map; the source may live outside the apply doc.

See the hot-reload example for a live demonstration.

Custom node types

The default registry covers intrinsics: container, sprite, nineSliceSprite, tilingSprite, animatedSprite, text, bitmapText, graphics, slot. Anything else — "type": "SpinButton", "type": "ProgressBar", "type": "spine" — needs a NodeType registered by the host.

Spine and other engine-specific objects are intentionally not built-ins. Register them as custom nodeTypes so your engine owns the Spine runtime dependency and field semantics.

The contract

interface NodeType {
    create(node: ResolvedNode, ctx: BuildContext): Container;
    assign?(node: ResolvedNode, target: Container, ctx: AssignContext): void;
}
  • create constructs an empty container subclass.
  • assign writes type-specific fields. Called by both build and apply — write your field-application logic once.

Example: SpinButton

import { build, defaultNodeTypes, type NodeType } from "pxd";
import { Container, Text } from "pixi.js";

class SpinButton extends Container {
    private caption = new Text({ text: "" });
    constructor() { super(); this.addChild(this.caption); this.eventMode = "static"; }
    setLabel(s: string) { this.caption.text = s; }
    setEnabled(on: boolean) { this.alpha = on ? 1 : 0.4; this.eventMode = on ? "static" : "none"; }
}

const SpinButtonType: NodeType = {
    create: () => new SpinButton(),
    assign: (node, target, ctx) => {
        if (!(target instanceof SpinButton)) return;
        if (typeof node.text === "string") target.setLabel(ctx.readString(node.text));
        if (typeof node.enabled === "boolean") target.setEnabled(node.enabled);
    },
};

build(doc, {
    resolve: { texture: Assets.get },
    nodeTypes: new Map([...defaultNodeTypes, ["SpinButton", SpinButtonType]]),
});

Custom fields live on the node

There is no props container. Custom scalar fields sit at the top level and participate in decision resolution.

{
    "id": "spinBtn",
    "type": "SpinButton",
    "x": 600, "y": 540,
    "text": { "_": "SPIN", "mobile": "TAP" },
    "enabled": true
}
Custom field validation is your responsibility. The library does no schema check on custom node fields beyond decision-map structural validation. Verify types and ranges inside NodeType.assign (use typeof / instanceof guards as in the example) or in a host-level pre-flight before calling build.

Custom nodes can have document-defined children

Custom/runtime nodes are composable. build() creates the custom node, runs its assign, applies base fields, then builds and adds its children. apply() walks children exactly like for container.

{
    "id": "panel",
    "type": "Panel",
    "title": "Menu",
    "children": [
        { "id": "caption", "type": "text", "text": "PLAY" }
    ]
}

Prefab references remain different: they are references to prefab bodies and cannot carry children on the reference node.

See custom-node-type and custom-composable-node for live demos.

Migration note: no props

Older drafts used { "props": { "text": "SPIN" } } for runtime/custom nodes. Custom parameters now live directly on the node: { "text": "SPIN" }. props is rejected so custom scalar fields can participate in the same decision-resolution pipeline as built-in fields.

Decisions and bindings

Two load-time resolution features let one document respond to host state without being rewritten:

FeatureWhat it doesWhere it applies
Decision values (§3.6)Pick a primitive based on the active tag set.Scalar fields such as x, texture, text, fill, and custom scalar fields.
String bindings (§7.2)Replace {path} placeholders through resolve.binding.Inside any string value after decisions are resolved.
 
{
    "id": "title",
    "type": "text",
    "text": { "_": "Welcome, {user.name}!", "ru": "Добро пожаловать, {user.name}!" },
    "x": { "_": 100, "mobile": 50 },
    "maxWidth": { "_": 320, "de": 400, "de+mobile": 360 }
}
const root = build(doc, {
    activeTags: ["de", "mobile"],
    resolve: {
        texture: Assets.get,
        binding: (path) => path === "user.name" ? "Ada" : "",
    },
});

// Later: same Containers, different resolved values.
apply(doc, root, { activeTags: ["mobile", "dark", "en"], resolve });

Selector rules

  • Every decision map must have _ as the default.
  • Selectors are single tags (mobile) or lexicographically sorted tag sets joined by + (de+mobile, not mobile+de).
  • The most specific matching selector wins; declaration order breaks ties.
  • Structural fields are not decidable: id, type, mask, children, extensions, points.

Binding escapes

  • \{ → literal {
  • \\ → literal \
  • Substituted values are not rescanned for more bindings.
  • If resolve.binding is omitted, placeholders pass through literally.

See the decisions and bindings example.

Prefabs

A prefab is a reusable subtree declared once and instantiated by type. Documents with prefabs are Library-level documents. Each prefab instance gets its own id scope (§13.2), so internal ids do not collide between instances.

{
    "format": "pxd",
    "version": 1,
    "level": "library",
    "prefabs": {
        "Button.primary": {
            "id": "root",
            "type": "container",
            "children": [
                { "id": "bg", "type": "sprite", "texture": "btn_primary" },
                { "id": "caption", "type": "text", "text": "" }
            ]
        }
    },
    "root": {
        "id": "menu",
        "type": "container",
        "children": [
            { "id": "playBtn", "type": "Button.primary", "x": 100, "y": 50 },
            { "id": "settingsBtn", "type": "Button.primary", "x": 100, "y": 120 }
        ]
    }
}

Rules and restrictions

  • Prefab graphs must be acyclic; cycles are rejected by validation.
  • Prefab references cannot carry children. Their structure comes from the prefab body.
  • If a runtime nodeTypes map and prefabs both define the same type key, build rejects the collision.
  • Use apply() after build to patch specific prefab instance descendants by label path, e.g. playBtn.caption.
Use a prefab when…Use a custom node type when…
The widget is just a composition of intrinsic/custom nodes.You need actual class behaviour, event listeners, methods, or internal state.
Designers should edit the structure as JSON.The host engine owns construction and field validation.
Every instance is structurally identical.Instances need runtime-specific construction.

See the prefabs example.

Slots

A slot node (§4.5) is a passive named mount point: an empty Pixi Container tagged with Symbol.for("pxd.slot"). Your game mounts external content into it after build.

{
    "id": "hud",
    "type": "container",
    "children": [
        {
            "id": "reelsMount",
            "type": "slot",
            "slot": "Reels",
            "x": 100, "y": 200,
            "width": 800, "height": 480
        }
    ]
}
const root = build(doc, { resolve });
const reels = new ReelsContainer({ rows: 5, cols: 5 });
mountSlot(root, "Reels", reels);

const mount = getSlot(root, "Reels");
if (mount) mount.addChild(reelsOverlay);

Why slots and not labels?

  • find(root, "reels") uses Container.label; slots use a symbol tag and a semantic slot name.
  • Slot lookup survives label collisions and cross-module boundaries.
  • The slot field is a normal string field, so bindings work: "slot": "Board.{layout.type}".
  • Mounted content survives apply() because the host owns slot children.

Slots do not auto-size mounted content. If you declare width / height, read those values from the slot and scale your host-owned content yourself.

See the slots example.

CLI

The package exposes a pxd binary.

pxd validate path/to/document.json
pxd inspect path/to/document.json

pxd validate parses the JSON, checks pxd.schema.json for structural errors, then runs the semantic validator (validate()) for rules such as duplicate ids, mask resolution, prefab cycles, and extension requirements.

pxd inspect validates the document first, then prints shape, node/type counts, prefab counts, and a short tree. It does not build Pixi objects or resolve textures/styles.

What's not included

These are intentionally absent. They are not promised in any current version and are not partially shipped under a hidden flag.

  • No Scene level. Scene-shape documents are rejected with a clear error.
  • No apply full-reset / reconcile mode. apply is patch-only. A future mode: "full" is on the roadmap; do not depend on it.
  • No add/remove of children by apply. Structural changes require a rebuild.
  • No extension handler API (§9.2). Per-node extensions payloads are silently ignored. Document-level extensions are not loaded.
  • No reactivity. Bindings resolve once per build / apply call; call apply again when state changes.
  • No tree → PXD serialization. One-way only.
  • No built-in Spine, button, or engine widgets. Register them as custom nodeTypes.
